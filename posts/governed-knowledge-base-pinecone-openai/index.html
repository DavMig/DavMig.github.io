From Documents to Governed Retrieval: A Pinecone + OpenAI Implementation Journal

2026-02-26
3 minute read
Engineering , AI

From Documents to Governed Retrieval

When I first started this project, the goal was deceptively simple: update rule documents and make sure the system used the latest information. But very quickly, it became clear that “simple” was not enough. I needed a way to ingest new content while keeping every step traceable, auditable, and safe.

The system needed to do more than just store documents. It had to guarantee:

  • Safe versioning and promotion
  • Complete audit trails
  • Instant rollback if something went wrong
  • Explicit control over what goes live
  • Flexibility in embedding providers, using OpenAI in-app instead of relying on Pinecone

Why Governance Matters

In production, a naïve “upload and overwrite” approach can be disastrous. One misplaced update, and suddenly your retrieval system is serving incomplete or incorrect information.

I designed a process that separates preparation, processing, and promotion. Each step is deliberate, predictable, and safe:

  1. Prepare a new version without touching the live system
  2. Process all documents thoroughly
  3. Promote explicitly when everything is ready
  4. Rollback instantly if needed

To make this work, each component has a clear role:

  • Supabase manages governance state
  • Supabase Storage holds the original documents
  • OpenAI generates embeddings
  • Pinecone stores the vectors

Promotion is just a logical pointer flip — raw files remain untouched — keeping the system safe and auditable.

Structuring the Knowledge Base

To make governance first-class, I created a dedicated schema in Supabase:

  • kb_collections: groups related domains of documents (e.g., rules-official)
  • kb_versions: immutable snapshots tracking draft, processing, ready, promoted, or failed states
  • kb_assets: metadata and checksums for each uploaded file
  • kb_chunks: extracted text chunks with processing status
  • kb_audit_events: a full timeline of operational actions
  • kb_promotions: records of promotions and rollbacks

This ensures every action is traceable, every version accounted for, and every decision documented.

The Ingestion Pipeline

The pipeline is designed to be predictable yet flexible. Each step is an opportunity to maintain control and visibility:

  1. Admin creates a new draft version
  2. Documents are uploaded to Supabase Storage
  3. Text is extracted from txt, md, json, or pdf formats
  4. Text is chunked (CHUNK_SIZE=1200, CHUNK_OVERLAP=200 tokens)
  5. Embeddings are generated using OpenAI text-embedding-3-small
  6. Vectors are upserted into Pinecone with associated metadata
  7. The version transitions to ready

Generating embeddings in-app keeps the system provider-agnostic and flexible for future changes.

Promotion and Rollback

Promotion is a deliberate act, separated from processing. Only when a version is ready does it go live:

  1. Validate that the version is ready
  2. Mark it as promoted
  3. Update kb_collections.active_version_id
  4. Record the promotion in the audit log

Rollback is equally simple: a pointer switch in Supabase instantly restores the previous version. Safe, predictable, and transparent.

Versioned Namespaces in Pinecone

To prevent accidental overwrites, each collection uses versioned namespaces:

<collection_slug>__<version_id>

This approach allows multiple versions to coexist, isolates changes during processing, and makes switching between versions trivial.

Operational Considerations

The system requires a few environment variables:

  • OPENAI_API_KEY
  • PINECONE_API_KEY
  • PINECONE_INDEX_HOST

Optional tuning parameters help manage load and debugging:

  • KB_EMBEDDING_BATCH_SIZE
  • KB_EMBEDDING_MAX_RETRIES
  • KB_EMBEDDING_RETRY_BASE_DELAY_MS
  • INFERENCE_DEBUG=true for provider-level insights

These controls make the system robust under real-world conditions.

Features Delivered

The admin interface now supports a complete lifecycle for document updates:

  • Create and manage collections
  • Draft and process new versions
  • Upload documents
  • Trigger processing
  • Promote ready versions
  • Rollback if necessary
  • View full audit trails

This is not just ingestion; it is governed retrieval lifecycle management designed for safety and reliability.

Lessons Learned

Throughout the project, several key lessons emerged:

  1. Governance must be built-in, not bolted on after the fact
  2. Promotions should be logical, non-destructive operations
  3. Versioned namespaces simplify rollback and reduce risk
  4. Provider errors must be visible and actionable, not hidden in ambiguity
  5. Discipline in infrastructure and processes pays off, even for solo projects

Next Steps

Planned enhancements include:

  1. Retrieval-side filtering and version-specific evaluation metrics
  2. Automated validation checks before promotion
  3. Cleanup policies for archived namespaces
  4. Admin-configurable chunking and embedding strategies

Closing Thoughts

By introducing versioned governance, controlled promotion, and auditable operations, this project transforms simple document updates into a safe, repeatable, and predictable process. The system supports:

  • Operational safety
  • Reproducibility
  • Auditability
  • Instant rollback
  • Explicit control over AI costs and reliability

It provides a solid foundation for production-grade RAG operations, ready to handle real-world AI retrieval workflows.
