Objective: Architecting a High-Reliability GenAI Pipeline

The goal of this article is to detail the methodology and reasoning behind implementing a generative AI treatment for data extraction from raw pixels.

The use case is very concrete: I run a Trading Card Game (TCG) website with about 14,000 monthly visitors. I needed to automate the extraction of data from card illustrations—specifically, the illustrators’ names, which are not readily available in any database. While my site is small, it serves as my personal Engineering Laboratory for experimenting with “Enterprise-grade” technology integrations.

My primary drivers for this project were:

  1. Cost Control: Multimodal processing can be expensive and goal is to stay in free-tier.
  2. Confidence Rate: I need a reliability score near 100% because I’m running this website alone and I don’t have time to review thousands of cards.
  3. Professional Governance: Moving away from “shadow AI” toward versioned, trackable, and auditable prompt management, because this is how you do it.

The Control Tower: Governance & Tooling

To make this a serious, enterprise-level project, I built a dedicated GenAI Dashboard within my site’s administration panel.

  • Professional Prompt Management: Prompts are stored in the database, versioned, and tracked.
  • Observability: Full visibility into every AI request to monitor quality vs. cost.

The Iterative Process: Trial & Error

I began my tests with a sample of 10 cards using the Mistral AI API (specifically the ministral-3b-latest model for cost optimization). Since I already had some artist names in my DB, I injected them into the prompt context to help the model, rather than building a complex RAG system, pragamatism at its finest here !

Iteration 3: The Separator Bug

The Diagnostic: After some digging I was using commas to separate the artist list in my prompt. However, some artist names contained commas. This caused a logic collision. The Fix: I implemented a new logical separator that would never appear in a name and reran the test.

1function pickArtistSeparator(artists: string[]): string {
2  const candidates = ['%%', '||', '##', '@@', '~~']
3  for (const candidate of candidates) {
4    if (!artists.some((artist) => artist.includes(candidate))) {
5      return candidate
6    }
7  }
8  return '|'
9}

The Result: The confidence score actually dropped, which was ironically a good thing. It became more realistic. The model was finally admitting when it couldn’t read a card instead of hallucinating. But it still wasn’t usable for production.

Iteration 4: Signal vs. Noise (The Crop)

UNtil now the LLM was analyzing the entire card. The first rule of LLM efficiency is to reduce input noise. So let’s do it !

Iteration 5: The “Eureka” Moment (Cognitive Alignment)

Even with the crop, the vertical text was still a struggle. LLMs are trained on standard documents—horizontal text, so why not rotate it so that it could be more efficient, tokenized in a more friendly way for the LLM

I decided to apply a 90-degree rotation to the cropped image to present the text horizontally.

1  const cropped = await image
2    .extract({ left, top, width: cropWidth, height: cropHeight })
3    .rotate(90)
4    .png()
5    .toBuffer()

The Final Result: 100% Reliability. By rotating the image, the model’s native training kicked in perfectly. On my test dataset, I achieved a 100% success rate with high veracity.

Conclusion

We moved from a 20% success rate to 100% through a disciplined engineering approach: reducing noise, managing data collisions, and aligning the data format with the model’s strengths. One has to understand also how LLMs work, the tokenizer , etc.. to harness the small things that can really help improve the outcome.

The next step is to patch the pipeline for batch processing across thousands of cards.
