From PDF Text to Retrieval-Ready Chunks: A Deterministic Parsing Journal

2026-03-08
12 minute read
Engineering , AI
AI , RAG , PDF , pdfjs-dist , TypeScript , Retrieval , Parsing , Architecture

From PDF Text to Retrieval-Ready Chunks

This project did not start as a parser research exercise. It started with a practical need: take technical PDF documents and make them usable for retrieval.

That sounds straightforward until you actually try it.

At first glance, the task seems simple:

  • extract text from a PDF
  • convert it to Markdown
  • split it into chunks
  • index it

In practice, none of those steps are trivial if the goal is deterministic, maintainable, and good enough for production RAG.

The real challenge was not “getting text out of a PDF”. The real challenge was reconstructing enough of the document’s logic so retrieval would not be noisy, redundant, or misleading.

This article documents that journey:

  • what we tried first
  • what broke
  • which heuristics were too naive
  • how the pipeline evolved
  • and why the final design now has three distinct layers:
    • LogicalDocument
    • LogicalDocumentStructure
    • PassageChunk[]

The Original Goal

The initial goal was to create a TypeScript-only library that could turn text-based PDFs into something usable for RAG.

Constraints were strict:

  • TypeScript only
  • no OCR
  • no Python
  • no external APIs
  • no LLM parsing
  • deterministic heuristics only

That immediately ruled out the usual fallback strategy of “if parsing gets hard, ask a model to interpret the structure”.

So the parser had to stand on geometry and text patterns alone.

Why Raw PDF Text Is Not Enough

A raw PDF text layer is not a document model.

What extraction libraries usually give you is closer to this:

  • small text items
  • coordinates
  • font metrics
  • sometimes end-of-line hints
  • sometimes grouping, sometimes not

That means the parser must reconstruct, in order:

  1. lines
  2. paragraphs
  3. repeated artifacts
  4. headings
  5. section hierarchy
  6. retrieval units

If any of those steps is weak, retrieval quality degrades quickly.

A chunk that starts in the wrong place, inherits the wrong title, or duplicates too much context can still “look fine” in debug output while performing badly in real search.

First Attempt: Start From pdf2json

The first version of the pipeline was built around pdf2json.

The initial idea was reasonable:

  • pdf2json already gives coordinates and style metadata
  • the output is structured JSON
  • that should make reconstruction easier

The original target API looked something like this:

1parsePdfJsonToMarkdown(pdfJson: Pdf2JsonOutput): string

And the early pipeline was:

  1. read Pages[].Texts[]
  2. merge R[].T
  3. group items by y
  4. rebuild lines
  5. detect headings
  6. rebuild paragraphs
  7. emit Markdown

This worked just enough to prove the direction.

But the output was unstable on real rulebooks.

The First Major Problem: Paragraphs Are Harder Than They Look

One of the first obvious failures was paragraph reconstruction.

A naive rule like:

  • “if the previous line ends with punctuation, start a new paragraph”

sounds reasonable.

It was wrong.

In many technical PDFs, a sentence ends at the right edge of a visual line simply because of layout, not because a new paragraph starts. That led to outputs like this:

1# 1-2-1.When either playerisdefeated,the gameends. When one opponentis
2
3defeated, the opponent who was not defeated wins the game.

This exposed two different problems at once:

  • paragraph splitting was too aggressive
  • spacing reconstruction was weak

We gradually replaced the early rules with stronger ones:

  • vertical gap matters
  • indentation matters
  • right-edge fill matters
  • punctuation matters, but only as one signal among others

That was the first big lesson:

A PDF paragraph is not a sentence. It is a layout pattern.

Spacing Reconstruction Was Also a Trap

Another failure mode was glued words.

Typical examples looked like:

  • UpdatedJanuary
  • Attackingand
  • Ifeitherplayerfulfillstheconditionsfordefeat

This was not a “structure” bug. It was a text assembly bug.

The issue came from the fact that PDF extraction often returns adjacent text items without explicit spaces. The parser must infer when a space should exist from geometry.

A simplified version of the spacing logic looked like this:

1const currentItemEndX = item.transform[4] + item.width;
2const nextItemStartX = nextItem.transform[4];
3
4if (nextItemStartX - currentItemEndX > -0.5) {
5  pageTextBuffer += ' ';
6}

That helped, but it was still too brittle.

Eventually the spacing heuristics became more permissive between alphanumeric tokens, because many real documents had tiny overlaps or almost-zero gaps that still semantically implied a space.

This was another recurring pattern during the project:

Many “formatting bugs” were actually geometry tolerance bugs.

Why We Switched Focus to pdfjs-dist

After enough friction with pdf2json, we added a dedicated debug mode using pdfjs-dist.

That changed the workflow.

Instead of trying to trust the Markdown output directly, we added a raw debug path that exposed:

  • extracted text
  • line starts
  • positions
  • inferred block kind
  • inferred level

That gave us visibility into what the parser actually believed.

The pdfjs-dist extraction setup looked like this:

1const textContent = await page.getTextContent({
2  normalizeWhitespace: false,
3  disableCombineTextItems: true
4})

Then we reconstructed raw text item by item, respecting hasEOL and inferred spacing.

This was critical, because it let us inspect not just the final Markdown, but the geometry behind it.

The Breakthrough: Use X Position as a Structural Signal

One of the biggest improvements came from stepping back and looking at page layout visually.

The insight was simple:

  • if x shifts right, it often indicates deeper structure
  • if x shifts left, it often closes a deeper structure
  • multi-line continuations matter
  • but not every indent means “new heading”

That led to a debug format like:

1[[block=block-0028 kind=paragraph level=6 page=1 lines=2 startX=148.94 minX=148.94 maxX=184.34]] ...

At that point, we started classifying blocks from layout patterns, not just text patterns.

For example:

  • a line followed by more-indented continuation lines was likely a paragraph
  • a short single line at a stable indentation level was often a heading
  • a centered digit near page bottom was probably a page number (meta)
  • a repeated title on early pages was likely front matter

This layout-first approach was much more reliable than trying to infer everything from numbering syntax.

A Key Realization: Never Trust Numbering Alone

At several points, there was a temptation to rely on numbering depth:

  • 1.
  • 1-2.
  • 1-2-3.
  • 1-2-3-1.

That was rejected on purpose.

Why?

Because numbering is document-specific, and the goal was to stay agnostic.

A line like:

11 - 2 - 2 - 1. When either player receives battle damage...

is not necessarily a heading. It may just be a numbered rule paragraph.

That distinction mattered a lot, because if we turned every deep numbered clause into a heading, the tree became artificially fragmented. If we turned too many of them into plain text, the tree became too flat.

The solution was not “better regex”. The solution was:

  • geometry first
  • continuity second
  • structure validation third

The “Leaf Header” Problem

One heuristic we debated was:

If a header has no child, it probably is not a real header.

That turned out to be mostly right.

For example:

  • a header with no descendants
  • no real content boundary
  • and no structural children

was often just a misclassified paragraph.

But this rule could not be absolute. A true section can be a leaf. So the actual implementation became more careful:

  • provisional classification first
  • structural validation second
  • reclassification if the supposed header does not open any real child structure

This was one of the points where the parser became much more coherent.

Front Matter and TOC Were Polluting the Tree

Once we had a LogicalDocumentStructure, a new problem emerged.

The tree often contained duplicate roots such as:

  • one “Game Overview” from the Table of Contents
  • one real “Game Overview” in the body

That polluted:

  • hierarchy
  • paths
  • future retrieval
  • chunk generation

A typical bad case looked like:

  • sec-0001 = "1) Game Overview" from page 1 TOC/front matter
  • sec-0002 = "1) Game Overview" from the actual document

So we added a filtering pass after tree construction.

The filtering logic grouped sections by normalized title and removed likely non-documentary duplicates using signals such as:

  • very early page
  • same title reappearing later
  • poor content quality
  • TOC/front matter keywords
  • missing hierarchy where the later duplicate clearly had children

This was another important architectural decision:

Build the full tree first. Clean the tree second.

That separation made the logic much easier to reason about.

From Blocks to a Logical Document

At that point, the parser no longer emitted only Markdown. It had evolved into a multi-layer model.

1. Block

A reconstructed textual unit:

  • title
  • header
  • paragraph
  • list_item
  • meta

2. LogicalDocument

A flat, ordered document model containing:

  • document id
  • source
  • inferred title/version
  • all blocks

3. LogicalDocumentStructure

A hierarchical tree built from title and header blocks.

That produced SectionNode instances like:

 1type SectionNode = {
 2  id: string;
 3  docId: string;
 4  title: string;
 5  level: number;
 6  parentSectionId?: string;
 7  childSectionIds: string[];
 8  pathIds: string[];
 9  pathTitles: string[];
10  pageStart: number;
11  pageEnd: number;
12  blockIds: string[];
13  text: string;
14};

This was the moment when the project stopped being “PDF to Markdown” and became “PDF to logical document model”.

Why Section Trees Alone Were Still Not Enough

Once the section tree existed, a new issue became obvious.

Some sections were perfect retrieval units. Others were far too large.

For example:

  • a small section with one rule and one example could work as-is
  • a big section containing several sub-rules was too broad and too heterogeneous

That led to a new layer:

 1type PassageChunk = {
 2  id: string;
 3  docId: string;
 4  sectionId: string;
 5  pathIds: string[];
 6  pathTitles: string[];
 7  chunkIndex: number;
 8  text: string;
 9  pageStart: number;
10  pageEnd: number;
11  sourceBlockIds: string[];
12  kind: "section" | "paragraph" | "mixed";
13};

This was not just chunking for size. It was chunking for retrieval semantics.

Why We Kept Two Kinds of Chunks

A key design decision was to keep both:

  • one section chunk per section
  • finer paragraph / mixed chunks for detailed retrieval

This mattered because not all queries are local.

Some queries are broad:

  • “what are the fundamental game rules?”

Others are narrow:

  • “what happens when an EX Resource is used to pay a cost?”

If only broad chunks exist, precision suffers. If only tiny chunks exist, high-level retrieval suffers.

So the chunker now explicitly serves both purposes.

The First Version of Passage Chunking Was Too Redundant

The first implementation of PassageChunk was correct in spirit but weak in detail.

It preserved section context, but often repeated too much of the wrong context.

A representative problem looked like this:

  • chunk-001 and chunk-002 both started with the same earlier clause
  • later chunks repeated an outdated local heading
  • retrieval candidates became near-duplicates

This produced chunks that were technically valid but semantically sloppy.

The root cause was that chunking still started from section size, not from local logical units.

That was the next major shift.

The Critical Improvement: Split by Logical Unit First, Size Second

The chunker now follows a stronger rule:

First detect local logical units. Then apply size constraints.

This changed everything.

Instead of saying:

  • “accumulate blocks until we exceed 900 characters”

the chunker now says:

  1. detect local starts
  2. build local passage units
  3. only then aggregate by size if needed

Local starts can be:

  • owned header blocks
  • owned title blocks
  • structured text blocks that already behave like autonomous clauses

So if a section contains:

  • 1 - 3 - 2 - 1 ...
  • 1 - 3 - 2 - 2 ...
  • 1 - 3 - 3 ...

the chunker should start new local units on those boundaries, even if those blocks are still paragraph blocks.

That avoids this kind of bad output:

1chunk-002:
21 - 3 - 2 - 1 ... 1 - 3 - 2 - 2 ...

and replaces it with:

  • one chunk starting at 1 - 3 - 2 - 1
  • another chunk starting at 1 - 3 - 2 - 2
  • another chunk starting at 1 - 3 - 3

That was a major quality improvement.

Why Headers Do Not Count Toward Size

One subtle but important rule is that headers provide context, but do not count toward chunk size.

Without that rule, long technical headings would distort chunk boundaries and create many underfilled chunks.

So the current model is:

  • header contributes context
  • paragraph and list_item contribute payload
  • chunk size is measured on payload only

That keeps chunks informative without making them too title-heavy.

Why Full pathTitles Are Not Injected Into Chunk Text

Another issue we discovered is that many rulebooks use sentence-like headings.

That means pathTitles can become extremely verbose, for example:

  • top-level section title
  • subsection title
  • rule title
  • clause title

If the entire chain is injected into every chunk’s text, embeddings become noisy and repetitive.

So the design is now split:

  • full pathTitles stay in metadata
  • only the most local useful title is injected into chunk text

This preserves context without overwhelming the embedding signal.

Section Chunks That Were Too Empty

A separate issue appeared with very short section chunks such as:

  • 2) Card Information

That alone is too poor to be a useful embedding candidate.

The fix was not to remove section chunks altogether. The fix was to enrich weak section overviews with a deterministic preview of child titles.

So instead of:

12) Card Information

the section chunk can become:

12) Card Information
2
32 - 1. Card Types
42 - 2. Card Text
52 - 3. Traits

That makes overview chunks much more useful for broad retrieval.

Example of the Final Chunking Strategy

The final chunking logic can be summarized like this:

1buildPassageChunks(document, structure)

Phase 1: section overview

Always emit one section chunk:

  • broad
  • contextual
  • good for high-level retrieval

Phase 2: local unit detection

Scan the blocks owned by the section:

  • reset context on owned header / title
  • also start new local units on structured autonomous paragraphs

Phase 3: fine chunk emission

For each local unit:

  • emit one chunk if small enough
  • split only inside that local unit if necessary

This ordering matters.

It means the chunker does not blindly split a section by size. It splits a section by logic first.

Mermaid: The Final Retrieval-Oriented Flow

flowchart TD
  A[pdfjs-dist text extraction] --> B[Reconstruct lines]
  B --> C[Reconstruct blocks]
  C --> D[Build LogicalDocument]
  D --> E[Build raw section tree]
  E --> F[Filter non-documentary sections]
  F --> G[Build cleaned LogicalDocumentStructure]
  G --> H[Emit section chunk for each section]
  G --> I[Scan blocks owned by each section]
  I --> J[Detect local autonomous passage units]
  J --> K[Apply size only inside local units]
  K --> L[Emit paragraph or mixed passage chunks]
  H --> M[Final retrieval-ready chunk set]
  L --> M

Debugging Was Essential

None of this would have worked without dedicated debug outputs.

The admin playground became the main tool for iteration. It exposed:

  • raw extracted text
  • heuristic block reconstruction
  • logical document JSON
  • section tree JSON
  • passage chunk JSON

That made it possible to compare:

  • what the PDF visually showed
  • what the parser believed
  • what the chunker finally emitted

Without that debug surface, trial-and-error would have been much slower and much less reliable.

The Real Lessons

A few lessons emerged very clearly.

1. Deterministic PDF parsing is mostly about geometry

Not regex. Not sentence splitting. Not “AI interpretation”.

2. Debug views are not optional

If you cannot inspect intermediate states, you cannot improve heuristics safely.

3. Document structure and retrieval structure are different things

A section tree is not the same as an indexable chunk set.

4. Split by logic before size

Chunk size is not a primary structural signal.

5. Metadata should be richer than chunk text

Keep the full path in metadata. Keep chunk text semantically focused.

What Exists Now

The current system now supports:

  • deterministic extraction debugging with pdfjs-dist
  • logical block reconstruction
  • logical document modeling
  • section tree reconstruction
  • filtering of duplicate front matter / TOC sections
  • retrieval-oriented PassageChunk[]
  • section overview chunks
  • local passage chunks
  • admin playground inspection

This is already a solid foundation for a serious RAG ingestion pipeline.

What Still Needs Work

The system is much better than the first iterations, but not finished.

Future improvements could include:

  1. Better local clause detection beyond numbering-like patterns
  2. Better handling of examples vs rules vs notes
  3. Smarter section overview generation
  4. Better compaction of verbose path metadata for ranking
  5. Evaluation on a larger corpus of rulebooks/specifications

Closing Thoughts

What started as “parse a PDF and chunk it” ended up becoming a layered document modeling problem.

That was the real shift in thinking.

The final architecture is not:

  • PDF -> text -> chunks

It is:

  • PDF -> extracted text
  • extracted text -> logical blocks
  • logical blocks -> logical document
  • logical document -> section tree
  • section tree + blocks -> retrieval chunks

That extra modeling layer is what makes the system usable.

Without it, the retrieval pipeline would still technically work. But it would be noisier, less stable, and much harder to trust.

And that, in the end, was the whole point of doing it deterministically.
